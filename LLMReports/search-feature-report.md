# Search Feature Report

## Overview

There are two separate search systems in transpod:

1. **Dynamic word search** â€” database-driven, accessible at `/searchword`
2. **Static concordance pages** â€” pre-generated HTML, accessible at `/concordances`

---

## 1. Dynamic Word Search (`/searchword`)

### Database: `content/concordance.db`

SQLite file with 5 tables:

| Table | Key Columns | Purpose |
|-------|-------------|---------|
| `podcasts` | id, name, language | Podcast catalog |
| `episodes` | id, podcast_id, name | Episode names per podcast |
| `segments` | id, episode_id, start_time, end_time, text | Time-aligned transcript chunks |
| `words` | id, word, occurrences | Every indexed word with occurrence count |
| `entries` | word_id, segment_id | Junction table linking words â†” segments |

### Building the Index: `indexer.py`

- Iterates `content/` subdirectories, reads each podcast's `_config.md` for language
- Only indexes podcasts whose language appears in `INDEXLANGUAGES` (default: `["ja"]`)
- Skips already-indexed episodes â€” **incremental, not a full rebuild each run**
- Tokenizes Japanese using **sudachipy** in both C-mode (compound words) and A-mode (fine-grained morphemes), merging unique tokens; other languages use regex `\b\w+\b`
- Rejects segments where any word repeats more than `REPMAX` (default: 5) times consecutively â€” logs rejections to `rejected_segments.log`
- Caps each word at `REFMAX` (default: 1000) segment references

### Configuration: `indexer.config.json`

| Key | Default | Purpose |
|-----|---------|---------|
| `INDEXLANGUAGES` | `["ja"]` | Which podcast languages to index |
| `REFMAX` | `1000` | Max segment references per word |
| `REPMAX` | `5` | Max consecutive word repetitions before rejecting a segment |

To index non-Japanese content, add language codes (e.g. `"es"`, `"en"`) to `INDEXLANGUAGES`.

### Server Routes: `podserver.cjs`

**`POST /searchword`** (lines 1182â€“1284)
- 5-table JOIN query with `WHERE w.word = ? COLLATE NOCASE` (case-insensitive)
- Groups results by podcast â†’ episode â†’ segments
- Wraps matching words in `<mark>` tags for highlighting
- Sorts episodes by published date where available

**`GET /autocomplete`** (lines 1154â€“1176)
- Returns word suggestions as the user types
- Query: `SELECT word FROM words WHERE word LIKE ? ORDER BY length(word) LIMIT 10`

### Template: `views/searchword.hbs`

- Search form (POSTs to `/searchword`)
- JavaScript autocomplete dropdown (fetches from `/autocomplete`)
- Results rendered as nested `<details>` elements: podcast â†’ episode â†’ segments
- Each segment shows time range as a clickable link: `/play/{pod}/{ep}?t={start_time}&from=search`

## 2. Static Concordance Pages (`/concordances`)

**Generated by:** `create_concordance.py`

- Produces static HTML files in `public/concordances/`
- Simpler tokenization (regex only, no morphological analysis)
- Includes Wiktionary lookup links per word
- Run manually: `python create_concordance.py <podcast_name> <num_occurrences> <lang>`

**Server route:** `podserver.cjs` lines 600â€“610, template `views/concordances.hbs`

## Keeping the Index Up to Date

The index is **incremental but manual**. Run after new episodes have been transcribed:

```bash
conda activate transpod
python indexer.py
```

**When to run:** After `getpodcasts.py` downloads and transcribes new episodes. The indexer reads the `.json` transcript files produced by `transcribefast.py`.

**Important:** `nightly.sh` does **not** call `indexer.py`, so new episodes will not appear in search results until the indexer is run manually.
